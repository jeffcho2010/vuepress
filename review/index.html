# 2.解析度求不跑版的區段，稿的長寬

解析度市佔率
1920 x 1080 11.55% 桌電
768 x 1024 5.87% 平版
1366 x 768 5.77% 筆電，可能橫跨部分平版
1280 x 720 3.51%
1440 x 900 3.15%
1280 x 800 1.83%

1. 768x1024： 稿雖然是平版的稿，其實是橫跨 768~1279 之間，不需要以平板 1024 的高去設計版面，可以只使用 600-700 左右的高度去設計想像即可，求安全下莊，如果多出一個功能完全只為了這個區段，意義較低
*建議不要使用 768x1024，改用 768x700 以下的數字

2. 1024： 最好不要用這個寬度去套用桌機設計，使用平版不跑版樣式安全下莊，用桌機設計非常有可能出現需要大幅調整的狀況，例如高流活動快訊，hover 也可以 1280 以上套用就好，用 1024 會發生平版不用 hover 樣式完全不一樣的狀況（比如卡片 hover 之後才出現文字，但平版以下直接現出文字），這時候使用 1024 當作 hover 的最小尺寸就會吃癟，需要額外處理
*建議 hover1280 以上即可，1024 的尺寸調整完全放棄

3. 1440 x 780？： 沒有什麼有建設性的意見，也知道展示漂亮考量，但始終覺得 1440x780 怪怪的
*好像應該用更多人用的解析度

# 3. 表單流程圖？功能？

  1. 伺服端驗證：目前表單只有製作客戶端驗證，並沒有製作伺服端驗證，安全性不足
    *需要找時間再開發，不能一直沿用舊的程式不改進

  2. 表單流程：目前一切的表單功能與中間狀態都是在做的期間依照我自身的邏輯與 UIUX 概念去處理，但是時常會發生做的途中才發現一定要處理的新狀況
    *EX：高流因為寄信只寄到同一個信箱，但需要判斷是哪一個表單送出的信，寄信的程式就需要大幅修改

  3. 對流程：如果出現預想之外的流程，會出現大麻煩，EX：幔室布緹，如果出現一個流程是原始狀況沒有，如果原始程式狀況加不了流程，即使做了一個模擬的程式流程，對方無法加流程也是沒有任何辦法，就會多出大量無法預期的來回時間，取得對方的後端程式去修改雖然會延長工作階段的開發時間，但是可預期的，並且出去即成品
    *做類似有流程性的設計前最好先對過

  4. 固定流程：目前也算是有一整套我們的固定流程，但需要清楚定義
    *我有體力的話再去列目前表單程式的整個狀態流程、驗證時機、需要出的設計稿狀態樣式

# 7. 後台介面 vs 前端畫面

前端畫面觀看者： 客戶、一般使用者
後台介面觀看者： 只有客戶

後台：一個只給客戶看以及使用的一種「前台」，相對前端的修改來說，性價比低很多 自製後台：可能可以實行，但也需要時間開發，近乎是另一個「前端」
資料庫 - 中間溝通程式 - 後台、前台
資料庫 <-> 中間溝通程式叫資料寫入資料庫
後台 <-> 中間溝通程式叫資料與寫入資料庫
前台 <-> 中間溝通程式叫資料，有時候有寫入資料庫
*建議對 wordpress 的後台要求降低一些，除非轉向自製後台除外

# 8. Icon

假設 16x16 的工作區域，圖的大小 12 x 12，左右各留 2，因為會等比放大，如果桌機 32px 手機 16px 的案例，左右的 2px 到了桌機就會變成 4px，間距就差了 2px，變成即使設計稿上左右 10px，變成手機版要寫 8px，桌機寫 6px

# 11. 字級設定

  1. 每一種字體的 line-height 狀況不一樣，EX：土地公的大標字體，如果特別去拉他在實際切版的時候也會有狀況，padding 都是假的
    *怪字體可能要先確認字體狀況，或是看實際網站呈現找到的字體使用狀況正不正常

  2. 計算 VS 目測對位 ，假設 line-height45，字級 30，相當於有額外的 padding 上下 7.5px，如果調成 1，padding 上下是 15px，實際 padding 就會寫 7.5px，如果有特別的怪字體比如英文甚至沒法這樣計算

  3. 多國語言似乎可以直接判斷某個狀況的 letter-spacing 過寬，直接設定字級，做的時候就可以稍微對一下多國語言狀態，加速流程
    *可以直接設定好多語言的字級設定，能在同一份稿上呈現更佳，直接打字 VS zeplin 樣式
