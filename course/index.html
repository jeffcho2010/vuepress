# 1. 菜單文字與結構全數都有差異

以新竹為例

以新竹為例，桌機菜單一種，手機版菜單一種，網站地圖一種，footer一種
叫資料的時候都需要規劃，每一種新的組合都需要撰寫程式排列順序，甚至要重叫

*所有的文字理應都來自後台，散落在各處非常難維護
*不應以「可以寫一次就好」的心態為常態去處理文案

# 2.分段內文

特別的內文形式：自行車內頁
需要確認編輯彈性，需要額外撰寫程式
限制若太多對使用者來說不易用，相對這種後台製作也比較麻煩
自行車隱私權
其實在後台非我們能控制的情形資料格式不宜太複雜，會有更多的溝通成本

*一般的內文形式，一行程式碼就完成了
*若網站重心並非在這些文章，可以評估必要性

# 3. 小裝飾

沒有難度可言，就是費工
除外會需要注意細節，比如螢幕拉大縮小裝飾會不會被切到或凸出去
越多裝飾越費工
土地公：同一種裝飾旋轉用到底
腳踏車：顏色多一種就是多一張圖
若是放在某一張圖上，有一種方式是出成大家都一樣大小，就不需要一直對位

*就是費時間寫位置，一種裝置一個
*裝飾種類的多寡會影響出圖需要的時間、整理分析的功夫、溝通對圖的成本

# 4. 主題顏色

沒有難度可言，但需要全面性觀察所有的版面情形
EX：幾種顏色、幾種形式、變背景色會不會改變字體顏色

*就是費時間分析，而且會影響程式處理方式，不會是單純「這邊寫這個顏色」

# 5. 不是一組的元素，對齊是惡夢

輪播按鈕
表單標籤

*做稿都要先思考誰跟誰是一組的

# 6. 手機版唯一滿版的區塊

從container凸出去

*其實還好，有手段推出去，只是偶爾會因為結構或功能出現麻煩，比如輪播

# 7. svg元素旋轉與一般css元素旋轉差異

svg內的元素需要特別指定變形中心，很硬的寫法
普通css元素不用，可以指定%數

*相對來說是工程面的小細節，若要方便旋轉需要出成剛剛好大小的圖，但就又會有前述的對位問題了

# 8. 有文字的圖衍生的麻煩：裝置、多國語言

字縮小需要重出圖
多國語言需要翻譯，變成要出不同語言的圖，維護困難

*以「客戶絕對會改」為前提下去考量，選擇要不要在圖片中放字，一旦需要維護修改就是大成本

# 9. 圖層分析

看到複雜背景，就要做全面分析，誰在誰前面
有時候會有不可能這樣堆疊的圖層，就需要溝通
分析必定會有疏漏

*就是麻煩，有時候則是會有不可能這樣堆疊的圖層

# 10. 大動畫內的文字

可以觀察很多動畫網站並不是很在意某些文字看不見
有些文字可以試圖配合文字並結合動畫
但「一堆文字」擺放在各處就變成要到處顧慮
反過來動畫要遷就文字

*減少在大動畫中的資訊，若客戶執意需要這些資訊呈現，應反過來考量移除動畫
